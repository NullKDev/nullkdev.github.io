# Internationalization (i18n)

The site is bilingual: **English (default)** and **Spanish**. There are two separate i18n mechanisms — one for UI strings, one for content.

## 1. UI Strings

### Where strings live

- `src/i18n/en.ts` — English (source of truth)
- `src/i18n/es.ts` — Spanish

Both export a flat object keyed by dot-notation strings:

```ts
// en.ts
export const en = {
  'nav.blog': 'Blog',
  'about.bio_p1': 'Hi! I\'m a <strong>mobile developer</strong> with over {yearsExp} years...',
}
```

### How strings are applied

`Layout.astro` inlines the full translation dictionaries as JSON and runs a tiny engine that:

1. Reads `localStorage.getItem('lang')` (defaults to `'en'` on first visit)
2. Walks the DOM looking for `data-i18n*` attributes
3. Replaces content with the matching translation key

### HTML attributes

| Attribute | Effect |
|---|---|
| `data-i18n="key"` | Sets `textContent` |
| `data-i18n-html="key"` | Sets `innerHTML` (use for strings with `<strong>` etc.) |
| `data-i18n-title="key"` | Sets `title` attribute |
| `data-i18n-aria="key"` | Sets `aria-label` attribute |
| `data-i18n-placeholder="key"` | Sets `placeholder` attribute |

### Variable interpolation

Strings can contain `{varName}` placeholders. To provide values, set `window.__i18nVars` before the i18n engine runs:

```astro
<script is:inline define:vars={{ yearsExp }}>
  window.__i18nVars = window.__i18nVars || {}
  window.__i18nVars.yearsExp = yearsExp
</script>
```

The engine replaces `{yearsExp}` in all translation strings automatically. Currently used in `about.astro` for dynamic experience years.

### Adding a new UI string

1. Add the key + English value to `src/i18n/en.ts`
2. Add the same key + Spanish value to `src/i18n/es.ts`
3. Use the attribute on the HTML element:

```astro
<p data-i18n="my.new.key">English fallback text</p>
```

The fallback text (the element's initial content) is shown during the brief moment before JS runs, so always write it in English.

### Language toggle

`LanguageToggle.astro` stores the selected language in `localStorage` under the key `lang`. On click it:
- Saves the new lang to localStorage
- On content pages (blog/projects/photos): **redirects** to the translated URL if one exists
- On static pages (about, tools, home): **applies i18n in place** via `window.__applyI18n(lang)`

## 2. Content Translation

Blog posts, projects, and photos use a **folder-based locale suffix** convention.

### Convention

```
src/content/blog/
├── my-post/          ← EN version (ID: my-post)
│   └── index.md
└── my-post.es/       ← ES version (ID: my-post.es)
    └── index.md
```

### Fallback behavior

If a post has no `.es` version, the EN version is shown when the user switches to Spanish. No broken page, no redirect loop.

### URL structure

| Locale | URL |
|---|---|
| English | `/blog/my-post` |
| Spanish | `/es/blog/my-post` |

The slug is always the **base slug** (without the `.es` suffix).

### Detecting translation existence

Use `hasTranslation(postId, 'es')` from `data-utils.ts` to check at build time. This drives the smart redirect in `LanguageToggle`.

### Meta tags on content pages

Content pages inject these meta tags so `LanguageToggle` knows what to do:

```astro
<meta name="post-locale" content="en" />
<meta name="post-slug" content="my-post" />
<meta name="post-section" content="blog" />
<meta name="post-has-es" content="true" />   <!-- only if ES exists -->
```

## Adding a New Translated Section

If you add a new section that needs `/es/*` routes:

1. Create `src/pages/es/your-section/[...id].astro`
2. Mirror the logic from `src/pages/es/blog/[...id].astro`
3. Add the corresponding `meta` tags to both EN and ES page files
4. Update `LanguageToggle.astro` if the redirect logic needs a new section type
